Propuesta Web Dexeus Mujer

Enfoque Zoopa para rediseño, replatforming y activacion digital sin continuidad de Drupal
17 marzo 2026 • Documento de propuesta • Referencias: analisis de dexeus.com y Mayo Clinic Health System

Resumen Ejecutivo

Problema central: Dexeus Mujer tiene una marca excepcional, datos de autoridad muy fuertes y una presencia multilingue avanzada, pero su ecosistema actual combina deuda tecnica, fragmentacion operativa y una experiencia digital menos clara de lo que exige su posicionamiento clinico.

Resultado esperado: construir una nueva web de referencia en salud femenina que convierta mejor, escale internacionalmente, simplifique la operacion editorial y permita a Zoopa gobernar experiencia, landings, SEO/GEO, datos e integraciones sobre una base moderna, sin mantener Drupal.

1
Punto de Partida
Decision base
La web actual combina autoridad real y buen contenido con problemas tecnicos claros: stack legacy, hreflang mal resuelto, ausencia de schema, exceso de assets, CTA principal poco visible y arquitectura de home mas editorial que orientada a triage y conversion.
La nueva web debe mantener los activos ganadores de Dexeus y adoptar las mejores practicas operativas de Mayo: triage claro, utility bar con accesos clave, templates consistentes, HTML semantico, performance, datos estructurados y flujos orientados a accion.
Dexeus no necesita parecerse a Mayo en tono o estetica; necesita apropiarse de sus patrones de claridad operativa y combinarlos con la especializacion, sensibilidad y autoridad de Dexeus Mujer: 1935, primer bebe FIV en Espana, 3.000+ nacimientos al ano, 20.000+ ninos nacidos por FIV y tasas de exito publicadas.

2
Vision de Producto Web
North star
Objetivo de producto: Convertir dexeusmujer.com en la plataforma digital de referencia para salud femenina en Europa del Sur, capaz de: - orientar rapidamente a cada perfil de paciente - captar demanda organica y de paid con landings eficientes - reforzar confianza clinica y emocional - integrar CRM y automatizacion comercial - escalar idiomas, servicios y contenidos sin rehacer el sistema
  • Homepage con triage claro por necesidades y perfiles, no solo por bloques editoriales.
  • Barra superior persistente con CTAs clave: pedir cita, area paciente, contacto / telefono, donacion si aplica.
  • Templates nativos para servicio, tratamiento, medico, unidad, localizacion, landing y articulo answer-first.
  • Base SEO/GEO avanzada: schema, FAQ, medicos, breadcrumbs, organization, medical organization, llms.txt y hreflang correcto.
  • Operativa de marketing preparada para HubSpot, formularios, scoring, landings y experimentacion.

Estructura Actual Verificada
Dominio + sitemap
La estructura pública actual no es un único site limpio. Es un ecosistema compuesto por site principal, blog separado, minisites satélite y secciones verticales. Además, el sitemap principal presenta una anomalía: los índices no segmentan correctamente por idioma y devuelven un universo global repetido.
La futura plataforma debe unificar experiencia, gobierno SEO y modelo editorial. Aunque algunos satélites puedan mantenerse por razones de negocio, el nuevo core debe tratarse como una arquitectura de plataforma y no como una simple sustitución de páginas.
Esquema real verificado a partir de robots.txt + sitemap.xml + cabeceras públicas: www.dexeus.com/robots.txt -> confirma stack Drupal legacy en el core site www.dexeus.com/sitemap.xml -> sitemapindex -> https://en.dexeus.com/sitemap.xml?page=1 -> https://en.dexeus.com/sitemap.xml?page=2 Importante: - ambos sitemaps contienen URLs de todos los idiomas y hosts - no están realmente segmentados por idioma - el índice principal devuelve el mismo universo global para cualquier host - el blog no aparece integrado en este sitemap principal

Colaboracion con la Agencia de Branding
Decision critica
Si la agencia de branding diseña cada pagina futura, el proyecto se vuelve lento, caro y dificil de operar. Si solo entrega identidad de alto nivel sin traduccion digital, Zoopa tendra que reinterpretar demasiado.
El mejor modelo es intermedio: la agencia define identidad y reglas, y Zoopa traduce eso a sistema digital, templates y ejecucion continua.
Modelo Fortalezas Debilidades Recomendacion
Libreria de elementos solamente Rapido, barato, deja libertad a producto Insuficiente para un site complejo; demasiada interpretacion posterior Valido solo si Zoopa asume toda la definicion UX posterior
Paginas tipo + libreria de componentes Da direccion visual real y permite sistematizar desarrollo Exige buena coordinacion entre branding y equipo digital Opcion recomendada
Diseno de todas las paginas por branding Control visual maximo a corto plazo No operativo, poco escalable, frena evoluciones y landings No recomendado
Zoopa diseña sistema y paginas sin agencia Maxima velocidad y coherencia digital Puede generar fricciones si la marca se redefine en paralelo Solo recomendable si branding ya esta cerrado
Modelo operativo propuesto branding + Zoopa: 1. La agencia de branding define: - principios de marca - tono visual - tipografia, color, iconografia - sistema fotografico / ilustrativo - reglas de composicion 2. Zoopa define y produce: - UX y arquitectura de informacion - design system digital - componentes funcionales - paginas tipo prioritarias - reglas responsive - templates CMS - landings y paginas evolutivas 3. Gobernanza compartida: - aprobacion de foundations por branding - aprobacion de experiencia y rendimiento por Zoopa - comité corto de decision para evitar bloqueos
La propuesta adjunta refuerza esta conclusion: lo operativo es trabajar con foundations de marca, componentes y paginas arquetipo. El diseno de todas las paginas futuras no escala y bloquea la evolucion de landings, SEO y crecimiento continuo.

Tecnologias de Implementacion
Comparativa
Opcion Descripcion Fortalezas Debilidades
WordPress nativo custom theme Frontend y CMS en WordPress con tema a medida, bloques/modulos custom y stack server-side Menor complejidad, mejor operativa editorial, coste controlado, SEO robusto, velocidad de salida alta Menos flexibilidad futura que un headless puro; performance depende mucho de buena implementacion

WordPress + HubSpot para landings separadas Site principal en WordPress y parte de las campañas en HubSpot CMS Agilidad para marketing y automatizacion directa Fragmenta marca, analitica, SEO y mantenimiento; duplica sistemas
Queda descartado mantener Drupal. Las dos rutas serias son: WordPress unico con desarrollo nativo a medida, o arquitectura desacoplada con frontend moderno y CMS estructurado. La eleccion depende del equilibrio que Dexeus quiera entre autonomia editorial clasica y ambicion de producto digital.
Valoracion de la propuesta adjunta: es especialmente solida en SEO tecnico, GEO, modelado de templates, estructura de contenidos y definicion de stack desacoplado. La incorporamos como referencia valida si Dexeus quiere maximizar rendimiento y escalabilidad, aunque para equipos con fuerte necesidad de autonomia editorial WordPress sigue siendo una opcion muy competitiva.
Ajuste derivado de los datos adjuntos: el site ya mueve aproximadamente 276.000 visitas mensuales y el plan SEO/GEO aspira a crecimiento adicional. La infraestructura objetivo no debe dimensionarse para el tráfico actual medio, sino para soportar campañas, picos de demanda y un escenario razonable de 450.000-700.000 sesiones/mes sin degradación perceptible.

Inferencia basada en datos disponibles: con 276k visitas/mes, 38.920 visitas mensuales a contacto y expectativa de crecimiento SEO, conviene diseñar la plataforma con un margen de al menos 2-3x sobre el tráfico medio actual. Eso permite absorber campañas, estacionalidad y expansión internacional sin redimensionados urgentes.

Recomendacion Tecnica Zoopa
Target stack
  • Descartar completamente Drupal del escenario futuro y planificar una migracion integral.
  • Elegir entre dos rutas viables: WordPress unico con desarrollo nativo a medida, o frontend desacoplado con Next.js / Astro y CMS estructurado.
  • Desarrollo basado en libreria de componentes, templates y modularidad editorial.
  • Multidioma de origen con arquitectura clara por idioma y gobierno de traducciones.
  • Integracion HubSpot para formularios, tracking, workflows, lead routing y medicion de conversion.
  • Landings en el mismo sistema visual y tecnico siempre que sea posible.
  • SEO/GEO by design: schema, FAQs, perfiles de medico, organization data, redirecciones y control semantico.
  • Analitica moderna: GA4, GTM, eventos, conversiones y mapas de atribucion alineados con CRM.
  • Busqueda avanzada multiidioma, FAQs estructuradas y perfiles de medico con datos enriquecidos.

Decisiones recomendadas

CMS objetivo WordPress unico o CMS headless; Drupal descartado
Modelo de diseño Sistema + templates + componentes
Gestion de landings Dentro del mismo stack, con modulos especificos
CRM / automatizacion HubSpot integrado, no aislado
Objetivo tecnico HTML semantico, schema completo, rendimiento premium

Observabilidad, Friccion y Quality Control
Operación continua
En un site sanitario de alto tráfico no basta con medir sesiones y formularios enviados. Hay que observar comportamiento, detectar caídas de rendimiento, aislar fricciones por idioma/servicio/dispositivo y convertir ese aprendizaje en mejoras continuas de conversión.
Capa de observación Sistema recomendado Qué permite ver
Analítica web GA4 + GTM + BigQuery Canales, sesiones, eventos, scroll, CTAs, formularios, call clicks, journeys y cohortes
CRM / lead analytics HubSpot Lifecycle, lead source, scoring, workflow performance, MQL-SQL-opportunity-patient
UX analytics Microsoft Clarity, Hotjar o Contentsquare Heatmaps, session replays, rage clicks, scroll depth, abandono de formularios
Call tracking CallTrackingMetrics u opción equivalente Campaña -> llamada -> lead -> cita, especialmente importante si el teléfono sigue siendo canal clave
Rendimiento técnico Sentry + Datadog/New Relic + uptime monitoring Errores JS, fallos de formularios, latencia, uptime, consumo, cuellos de botella y regresiones
Experimentación GrowthBook, VWO o AB Tasty Tests de CTA, formularios, copys, layouts y elementos de confianza
Bucle recomendado de depuración de fricción: 1. Detectar: - caída de CTR - caída de submit rate - aumento de abandono - empeoramiento de Core Web Vitals 2. Aislar: - idioma - servicio - dispositivo - canal - plantilla 3. Observar: - session replay - heatmap - field-level analytics - errores de formulario - logs técnicos 4. Hipótesis: - CTA poco visible - formulario demasiado largo - prueba de confianza insuficiente - paso de navegación ambiguo - carga lenta o error técnico 5. Corregir: - cambio de layout - simplificación de formulario - mejora de copy - fast track de contacto - optimización técnica 6. Validar: - test A/B - medición de uplift - decisión de despliegue o rollback
Punto de fricción Cómo lo detectamos Cómo lo corregimos
El usuario no ve el CTA Heatmaps, scroll depth, CTA CTR Reordenar hero, CTA sticky, bloques de confianza previos a la acción
El usuario abre formulario y no termina Form open rate vs submit rate, field analytics, replays Reducir campos, dividir pasos, añadir progresión, reforzar privacidad y expectativa
Mucho tráfico informacional pero poca captación GA4 content groups, pathing, assisted conversions CTAs contextuales, lead magnets, bloques doctor/centro/solución dentro del contenido
Caída en móvil Segmentación device, CWV mobile, rage clicks Reducir peso, mejorar sticky CTA, simplificar formularios y above-the-fold
Leads que no avanzan a cita HubSpot lifecycle, SLA comercial, análisis por fuente Lead routing, scoring, workflows, mejor handoff y seguimiento comercial
Quality control Cadencia Checklist mínimo
Pre-lanzamiento Una vez, antes de go-live SEO técnico, renders, formularios, tracking, consent mode, accesibilidad, QA multidioma, redirects, schema, sitemap
Semanal Operación continua Uptime, errores, submits, leads por fuente, campañas activas, páginas críticas
Mensual Comité de crecimiento Funnel, fricciones por servicio, landings, contenidos top, search, SEO/GEO, hipótesis CRO
Trimestral Dirección y roadmap Capacidad técnica, deuda, rendimiento por idioma, atribución, impacto de nuevos módulos y backlog evolutivo
En un entorno sanitario hay una condición adicional: la observación debe ser privacy-safe. Session replays y analytics deben enmascarar PII, respetar consentimiento y evitar registrar datos sensibles de salud en herramientas no preparadas para ello.
La captación no mejora solo con más tráfico. Mejora cuando marketing, UX, tecnología y CRM pueden ver exactamente dónde se rompen los journeys y corrigen esas roturas con ciclos cortos, medibles y repetibles.

Estimación de Diseño
Sistema visual + páginas tipo
La nueva web no debe diseñarse página a página. Debe diseñarse como un sistema visual y funcional capaz de producir cientos de URLs a partir de un número limitado de patrones. La agencia no debería preparar cientos de pantallas, sino un set preciso de foundations, componentes y páginas tipo.
Si la agencia intenta diseñar una por una las páginas futuras, el proyecto se encarece, ralentiza aprobaciones y compromete la escalabilidad de WordPress, de las landings y del roadmap evolutivo.
Elemento de diseño Volumen recomendado Qué debería cubrir
Foundations 1 sistema base completo Tipografía, color, grid, espaciado, radios, iconografía, fotografía, motion y criterios responsive
Componentes reutilizables 25-40 Héroes, CTAs, cards, formularios, FAQs, testimonios, métricas, listados, navegación, trust blocks y módulos de captación
Patrones de composición 8-12 Cómo se combinan los componentes en estructuras repetibles para páginas largas, cortas, listings y landings
Páginas tipo 12-18 Arquetipos completos listos para desarrollo y reutilización editorial
Variantes de landing 4-6 Paid, evergreen SEO, lead magnet, evento, internacional y tratamiento
Variantes de hero / CTA / formularios 8-12 Versiones para especialidad, médico, internacional, fertilidad, embarazo y campañas

Reparto recomendado de trabajo: Agencia de branding: - foundations visuales - principios de marca digital - sistema fotográfico / ilustrativo - iconografía - 3-5 page archetypes de referencia de máximo nivel Zoopa: - traducción de marca a design system digital - definición de 12-18 páginas tipo - creación de 25-40 componentes - reglas responsive - variantes para CMS - landings - evolución posterior del sistema
Escenario Volumen de diseño Valoración
Mínimo viable 10-12 páginas tipo + 20-25 componentes Posible, pero justo para un site tan rico
Recomendado 12-18 páginas tipo + 25-40 componentes Equilibrio correcto entre control y escalabilidad
Excesivo / poco operativo 25+ páginas tipo diseñadas una a una No recomendable; deriva en pseudo-maquetación manual
Conclusión: la agencia debería preparar aproximadamente un sistema visual completo, entre 25 y 40 componentes y entre 12 y 18 páginas tipo. Por encima de ese volumen deja de ser diseño de sistema y pasa a ser producción manual poco eficiente.

Recomendacion final: la mejor propuesta para Dexeus Mujer es una nueva plataforma web diseñada como sistema digital de crecimiento, no como un rediseño cosmetico. La agencia de branding deberia aportar foundations y algunas paginas tipo; Zoopa deberia liderar UX, design system digital, templates, implementacion y evolucion. Tras revisar la propuesta adjunta, dejamos consolidado que Drupal no debe mantenerse. Las dos rutas viables son un WordPress unico, desarrollado en nativo a medida, o una arquitectura desacoplada con frontend moderno y CMS estructurado. En ambos casos, HubSpot, multidioma, SEO/GEO y landings deben resolverse desde el nucleo del proyecto y no como capas posteriores.
